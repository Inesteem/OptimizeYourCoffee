COFFEE APP CODE REVIEW - EXECUTIVE SUMMARY

PROJECT: Raspberry Pi Touchscreen Coffee Sampling App
SIZE: 1,110 lines (app.py) + 5 templates + 2 JS files

DATE: 2026-04-02

KEY FINDINGS
============

CRITICAL ISSUES (Fix immediately):
1. DATABASE CONNECTION POOLING
   - Current: 28 new connections per request
   - Impact: Memory exhaustion, Pi slowdown, scalability blocker
   - Fix: Use flask.g pattern (10 min, 1 codeblock)

MAJOR ISSUES (Fix before adding features):
2. DRY VIOLATION IN FORM PARSING
   - Location: add_coffee() and save_coffee() routes (40+ identical lines)
   - Impact: Maintenance nightmare, bugs propagate to both places
   - Fix: Extract extract_coffee_form() helper (12 min)

3. GLOBAL STATE MUTATION
   - Location: render_tasting_notes() mutates TASTING_EMOJIS dict
   - Impact: Hidden side effects, breaks testing, mysterious bugs
   - Fix: Use .copy() to avoid mutation (3 min)

4. EMOJI MAPPING DRIFT
   - Python TASTING_EMOJIS: 48 entries
   - JavaScript NOTES dict: 60+ entries
   - Impact: Frontend/backend inconsistency, confusing UX
   - Fix: Create shared_data.json, load by both (20 min, next session)

5. HARDCODED OPTIONS IN TEMPLATES
   - Process options appear in: app.py, step1_coffee.html, edit_coffee.html
   - Impact: Changes require edits in 3 places, sync risk
   - Fix: Data-driven templates via context processor (5 min, with problem 4)

SECURITY ISSUES (Medium priority):
6. STORED XSS VULNERABILITY
   - Location: autocomplete.js line 76, tastingnotes.js line 148
   - Pattern: innerHTML with unsanitized DB data
   - Impact: If DB is compromised, XSS runs in browser
   - Fix: Replace innerHTML with textContent + DOM methods (10 min, next session)

QUESTIONS YOU ASKED - DIRECT ANSWERS
=====================================

Q1: Best way to split app.py without overengineering?
A: Functional split (not domain-driven design):
   - database.py (connections, init_db, backups)
   - utils.py (safe_int, safe_float, extract_coffee_form)
   - routes/ (coffee.py, sample.py, evaluation.py, stats.py, api.py)
   - app.py (factory + entry point)
   Only use this IF app hits 1,500 lines. Currently at 1,110.

Q2: Extract emoji mapping to JSON for Python/JS?
A: YES. Absolute yes. Create static/shared_data.json with emojis + processes.
   Python loads at startup: TASTING_EMOJIS = load_shared_data()["emojis"]
   JS fetches via /api/shared-data endpoint
   This solves problems 3, 4, and future data consistency issues.

Q3: DRY form parsing - helper vs WTForms/marshmallow?
A: Use a simple helper function (extract_coffee_form).
   WTForms/marshmallow are overkill for a Pi kiosk app.
   Simple helper keeps it lightweight, testable, easy to debug.

Q4: Architecture patterns for scaling (charts, recipes, ML)?
A: First: Fix DB pooling and split into modules.
   Then: Add /api routes that return JSON (no HTML overhead).
   Example: /api/stats/best-coffees?days=30 for charts
           /api/recipes/<coffee_id> for recipes
           /api/grind-recommendation/<coffee_id> for ML
   Keep business logic in separate functions, expose via routes.

Q5: Most impactful 30-minute refactoring?
A: Do Tier 1 (all three tasks above).
   In order:
   1. Fix DB pooling (biggest impact on stability)
   2. Extract form helper (biggest impact on maintainability)
   3. Fix global mutation (prevents future bugs)
